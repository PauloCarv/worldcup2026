# World Cup 2026 Live Tracker — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up the World Cup 2026 live-tracker site (imported from Claude Design) as a static site on GitHub Pages, then wire up a daily scheduled agent that keeps results/bracket/highlights current.

**Architecture:** Three plain files (`index.html`, `data.js`, `support.js`) served statically. The root page's `dc-import` of a separate `bracket-node` component is replaced with inlined markup + computed fields, because the dc-runtime's component registry only auto-populates for the page calling `boot()` directly — there is no built-in fetch-by-name for `dc-import` outside Claude's design-preview iframe (confirmed by reading `support.js`: only `x-import`, which takes an explicit URL, has fetch support). Inlining avoids depending on undocumented runtime internals while keeping the rest of the `x-dc`/`sc-for`/`sc-if`/`DCLogic` component model untouched.

**Tech Stack:** Static HTML/CSS/JS, Claude's `dc-runtime` (`support.js`, vendored, do not hand-edit), GitHub + GitHub Pages, a Claude Code scheduled cloud agent (`schedule` skill) for daily updates.

**Spec:** `docs/superpowers/specs/2026-06-24-worldcup2026-site-design.md`

### Task 4: Add `.nojekyll`

**Files:**
- Create: `.nojekyll`

GitHub Pages runs Jekyll by default, and Jekyll's Liquid templating engine also uses `{{ ... }}` syntax — it would try to interpret the page's own template expressions (`{{ tabGroupsStyle }}`, `{{ m.cardStyle }}`, etc.) as Liquid variables and silently blank them out. An empty `.nojekyll` file at the repo root disables this.

- [ ] **Step 1: Create the file**

```bash
touch .nojekyll
```

- [ ] **Step 2: Commit**

```bash
git add .nojekyll
git commit -m "Disable Jekyll processing for GitHub Pages"
```
